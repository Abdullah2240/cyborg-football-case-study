01 / The Brief

The client needed a custom game for a physical football-throwing cabinet.

Aethagon is building a coin-operated arcade cabinet where players throw real footballs at a touchscreen LCD. The hardware was in progress — IR touch frame, USB arcade buttons, dual HDMI displays — but they needed game software that could drive all of it reliably in a 24/7 venue setting.

The brief called for three distinct game modes with different pacing, an arcade-style Hall of Fame leaderboard, operator controls for volume and display configuration, and a visual identity strong enough to draw a crowd from across a venue. Every value that might need to be tuned later had to be configurable without touching code.

Most importantly: the game software had to integrate with physical hardware being built in parallel by a separate firmware team, without locking them into a specific wire protocol. The software needed to land first; the hardware team needed an integration contract they could wire against independently.

02 / My Role

Solo engineer, architect, and technical lead.

Game implementation

03 / Tech Stack

Chosen for reliability over novelty.

Python 3.11 Pygame CE 2.6 OpenCV pyserial pygame.joystick HID touch JSON persistence Windows 10/11

Why Pygame CE

Rock-solid 2D rendering, first-class Windows support, zero deployment overhead, and predictable frame pacing for a 60 fps fullscreen game. No launcher, no runtime, no licensing headache.

Why OpenCV

Pygame has no native video playback. OpenCV's VideoCapture decodes an MP4 frame-by-frame into Pygame surfaces for the attract-mode idle video — lightweight, codec-flexible, no GStreamer required.

Why pygame.joystick

Any USB HID gamepad encoder works out of the box with zero drivers. Hot-plug handled via JOYDEVICEADDED / JOYDEVICEREMOVED. Vendor-independent — the firmware team isn't locked into one board.

05 / Architecture

Built to be handed off, not just shipped.

11-state FSM

Each state is a BaseState subclass with on_enter / update / render / on_exit lifecycle hooks. The state machine routes transitions centrally and hands off data between states via stashed attributes — no state knows about its neighbours directly.

Hardware-agnostic input

Game code calls input_handler.get_events() and gets back a flat list of events. Three independent transport streams (IR touch, USB arcade encoder, dev keyboard) merge into one queue. No state file ever touches a wire protocol.

Config-driven

~320 lines of tunable constants in config.py. Timers, thresholds, colours, layout geometry, file paths, animation speeds — all in one place, inline-documented. Non-engineers on the client team can tune gameplay values without reading game code.

Graceful degradation

Missing images render as coloured placeholder shapes, missing audio plays silence, missing idle video is skipped with no error. You can run the game against a completely empty assets/ folder and it still boots — critical for partial asset deliveries during art production.

Frame-rate independent

All timers use delta time from clock.tick(FPS). Milestone celebrations, attract animations, zone shrink schedules, and transition countdowns all behave identically whether the cabinet runs at 30 fps or 120 fps.

Dual-display compositor

Two 1920×1080 render surfaces, composited each frame into one 3840×1080 fullscreen window. States receive both surfaces in their render() call and draw to each as needed. Single-display dev mode aliases both pointers to one surface.

06 / Challenges Solved

The interesting problems.

Shipping game code ahead of finished hardware

The game had to be code-complete and testable before the physical cabinet existed. I built a dev keyboard backend that simulates every event the real hardware would eventually send — coin inserts, target hits at predefined coordinates, operator menu access, reset. The entire game loop was exercisable from a laptop. When the firmware team later wired the real hardware, it plugged into the same event interface with zero game-code changes.

Supporting any USB arcade encoder, no drivers

Arcade button encoders fall into two camps — keyboard-emulating and gamepad-style. I went with the gamepad path because it keeps physical buttons cleanly separated from the dev keyboard namespace and avoids USB n-key-rollover limits. The handler supports discrete buttons via JOYBUTTONDOWN, direction sticks via JOYHATMOTION, and hot-plug recovery via JOYDEVICEADDED / JOYDEVICEREMOVED — unplug mid-game and the dev keyboard takes over, replug and the buttons come back.

Dual-display compositing with runtime toggle

The cabinet runs two physical LCDs as a single 3840×1080 fullscreen window. In dev, the same code runs on one 1920×1080 monitor with DUAL_DISPLAY=False. The rendering architecture uses two render surfaces; in single-display mode, both pointers alias to the same surface so state code doesn't branch on display mode. The operator can toggle at runtime via the operator menu.

Documentation as a first-class deliverable

The firmware team needed an integration contract, not a tutorial. I wrote the hardware integration guide peer-to-peer — interfaces, event semantics, calibration steps, edge cases, troubleshooting — assuming the reader knew USB HID, pygame.joystick, and Windows device management. Same standard for the asset delivery spec: exact filenames, formats, dimensions, fallback behaviour. Partial deliveries never block the build.
